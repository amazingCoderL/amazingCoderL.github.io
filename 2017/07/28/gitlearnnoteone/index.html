git使用（一）

发表于 2017-07-28 | 分类于 项目管理 |

git基本介绍

1. git是什么

git是一种分布式版本控制系统，并且是当前世界最流行，最先进的版本控制系统;所谓版本控制通俗讲就是git会帮你记录你的项目本次改动的记录,详细的定义不在此多介绍，零基础同学可以参考廖雪峰老师的教程。

2. git与svn的区别

大家知道git是一种分布式版本控制系统，svn是集中式，到底对用户来说有什么区别呢？

所谓集中式就是你所做的事情都集中在一个地方，也就是中央仓库，你要干活之前必须先从中央仓库获取你的项目，然后再自己的电脑干活，干完之后又推送到中央仓库,这样的话如果没有网或者或者中央服务器挂了你就没法工作了，这是集中式版本控制最大的缺点：无法单机，太过于依赖中央仓库。

什么是分布式，首先分布式没有中央仓库一说，每一台电脑都是自己的“中央仓库”，即只要你有一台电脑就能进行版本控制了，这样在没有网络的情况下，或者远程仓库挂了也不影响你的开发工作。

有人可能疑惑分布式的远程仓库和集中式的中央仓库不是一样吗？按功能来说确实一样，都是一个外网或者局域网的管理项目的服务器，但分布式的远程仓库只是用来共享项目的一个服务，没有它你同样可以进行版本控制，只是多人并行开发会受影响。

当然上面所述的优点只是git概念上的优势，在使用上git还有很强大的分支管理能力。

git的基本使用

git常用名词介绍

  • 本地仓库:个人电脑使用git初始化后的本地项目版本控制仓库(.git文件夹下)
  • 远程仓库:外网服务器或局域网服务器存放项目代码的仓库，可进行多人开发代码同步共享
  • 工作区:工作区通俗讲就是个人电脑硬盘里项目存放的文件夹（被git初始化过）
  • 暂存区:暂存区是git仓库里提供的一种新的区域(stage),暂存区通俗说就是工作区add和git库commit过程的一个中介区域，要想完成代码的commit操作，必须先将工作区内容add到暂存区才能commit，反过来说commit其实是提交的暂存区的修改，这个概念很重要。
  • 分支:初始化git库时会默认生成一个master，这就是一个分支

git命令行基本使用

1. 创建git仓库：
$ mkdir gittest
 $ cd gittest
 $ git init
 Initialized empty Git repository in /Users/hao/gittest/.git/

这时在gittest目录下会生成.git目录，git仓库就穿件成功了。

2. 修改内容提交到git本地仓库 git add/git commit -m “commit message”
$ git add readme.txt
$ git commit -m "abc commit"
[master (root-commit) b50ea8f] abc commit
 1 file changed, 1 insertion(+)
 create mode 100644 readme.txt
3. 查看当前分支的状态 git status
$ git status
On branch master
Initial commit
Untracked files:
  (use "git add <file>..." to include in what will be     committed)

readme.txt

nothing added to commit but untracked files present (use     "git add" to track)

这时可以看到提示readme.txt文本未提交更改

4. 查看修改文件的内容 git diff [filename]
$ git diff readme.txt
diff --git a/readme.txt b/readme.txt
index f2ba8f8..0c00383 100644
--- a/readme.txt
+++ b/readme.txt
@@ -1 +1 @@
-abc
\ No newline at end of file
+def
\ No newline at end of file

从上面信息可以看出我将readme.txt的内容从abc修改为def,再次提交修改的文件到仓库

$ git add readme.txt
$ git commit -m "def commit"
[master 306ecb9] def commit
 1 file changed, 1 insertion(+), 1 deletion(-)

查看分支状态

$ git status
On branch master
nothing to commit, working directory clean

此时工作区没有可提交修改了。

5. 查看版本信息 git log （追加–pretty=oneline查看简单信息）
$ git log
commit 306ecb90b1232474de9ed62a73dcadb2f63e8bab
Author: xxx <xxxxxxxxxxx@163.com>
Date:   Mon Jul 31 11:04:39 2017 +0800

def commit

commit b50ea8fe8418a16f2e90e7551411e2478b0953b7
Author: xxx <xxxxxxxxxxx@163.com>
Date:   Mon Jul 31 10:30:39 2017 +0800

abc commit

$ git log --pretty=oneline
306ecb90b1232474de9ed62a73dcadb2f63e8bab def commit
b50ea8fe8418a16f2e90e7551411e2478b0953b7 abc commit

上面的打印信息中commit后字符串为版本号，相当于一个版本的唯一识别码,和svn不一样，git的版本号不是一个有序的号码,通过log信息可以看到我总共提交了两次修改，现在一个产品妹子跑过来告诉你上次的需求不要了，这时我想回退到第一次提交怎么办？

6. 版本回退操作git reset –hard [HEAD^|HEAD~10]

中括号内为你需要回退到哪个版本，如上一个版本HEAD^,上上个版本HEAD^^,网上第十个版本HEAD~10.

$ git reset --hard HEAD^
HEAD is now at b50ea8f abc commit

查看版本信息

$ git log
commit b50ea8fe8418a16f2e90e7551411e2478b0953b7
Author: xxx <xxxxxxxxxxx@163.com>
Date:   Mon Jul 31 10:30:39 2017 +0800

abc commit

可以看到当前版本只有第一次提交的版本,这时那个妹子又来了，告诉你刚刚说的那个需求还是加上吧


这时如果只有几行代码你当然可以再写一遍，但是如果有上千行代码 你是不是要崩溃了，不急，我们能回到过去，也能穿越未来。找到刚刚提交def的commit id，

$ git reset --hard 306ecb
HEAD is now at 306ecb9 def commit
liuhaodeMacBook-Pro:gittest hao$ git log
commit 306ecb90b1232474de9ed62a73dcadb2f63e8bab
Author: xxx <xxxxxxxxxxx@163.com>
Date:   Mon Jul 31 11:04:39 2017 +0800

def commit

commit b50ea8fe8418a16f2e90e7551411e2478b0953b7
Author: xxx <xxxxxxxxxxx@163.com>
Date:   Mon Jul 31 10:30:39 2017 +0800

abc commit

commit id可以不写全也可以,如果刚刚你把终端关了或者电脑重启了无法找到提交def的commit id怎么办，可以这样：

$ git reflog
306ecb9 HEAD@{0}: reset: moving to 306ecb
b50ea8f HEAD@{1}: reset: moving to HEAD^
306ecb9 HEAD@{2}: commit: def commit
b50ea8f HEAD@{3}: commit (initial): abc commit

此时你可以看到你的每次操作记录，也可以查看每一次的commit id了。

7. 撤销修改操作 git checkout –[filename]

我现在对readme.txt添加了一句话，执行：

$ git status
On branch master
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in     working directory)

modified:   readme.txt

no changes added to commit (use "git add" and/or "git commit     -a")

此时提示有修改的内容未提交，项目开发时如果代码写一般发现有更好的设计思路，这时如果我改了上千行代码或者修改了好几个类想撤销了，可以这样操作

$ git diff HEAD -- readme.txt
diff --git a/readme.txt b/readme.txt
index 0c00383..cd4d08a 100644
--- a/readme.txt
+++ b/readme.txt
@@ -1 +1,2 @@
-def
\ No newline at end of file
+def
+我修改了readme
\ No newline at end of file

此时可以看到readme.txt文件的修改内容，然后撤销此次修改

$ git checkout -- readme.txt
$ git diff HEAD -- readme.txt

执行玩这两句发现没有任何提示，这说明撤销操作成功了，并且此时工作区的readme.txt文件和git仓库的文件没有区别.
如果此时我添加的这句修改已经被我手贱add到了暂存区，这时想撤销怎么操作,这时可以用上面介绍的git reset：

$ git add readme.txt
$ git status
On branch master
Changes to be committed:
  (use "git reset HEAD <file>..." to unstage)

modified:   readme.txt

$ git reset HEAD readme.txt
Unstaged changes after reset:
M    readme.txt
$ git status
On branch master
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git checkout -- <file>..." to discard changes in     working directory)

modified:   readme.txt

no changes added to commit (use "git add" and/or "git commit     -a")

从提示可以看到我把add到暂存区的修改撤销回了工作区。

8. 文件删除

工作区文件删除

$ rm readme.txt

git仓库文件删除,此时删除后需执行git commit -m “delete readme.txt”

$ git rm readme.txt
$ git commit -m "delete readme.txt"

文件误删恢复文件操作

$ git checkout -- readme.txt
