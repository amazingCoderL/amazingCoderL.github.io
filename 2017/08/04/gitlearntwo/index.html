git的使用（二）

发表于 2017-08-04 | 分类于 项目管理 |

远程仓库

这里还是用github作为自己的远程仓库，首先要有一个github账号，然后在自己的账号下创建一个Repository，具体怎样注册github，怎样创建Repository不在此详说了。

1. 生成SSH公钥

想将本地的git仓库与远程github仓库建立关联，需先生成你电脑的SSH key，这样github才知道是否是你提交的代码

$ cd .ssh
$ ls
id_rsa        id_rsa.pub    known_hosts

可以先看看自己是否生成过SSH公钥，即是否有id_rsa和id_rsa.pub文件，我这已经有了无需创建。

$ ssh-keygen -t rsa -C "youremail@example.com"

不出意外，执行完以上命令可以看到上面说的两个文件了

$ cat id_rsa.pub

此时会生成一长串字符,打开你的github 我的下面的Settings–SSH and GPG keys选项下面,点击New SSH key按钮，将你刚刚生成的公钥串填入确定即可。

2. 关联远程仓库

我现在本地有一个git仓库了，关于怎么创建本地仓库等问题，可以看我上一篇文章git的使用（一）,需要将本地仓库与远程仓库建立关联，比如你要将自己的牛皮项目开源，可以这样操作：

$ git remote add origin git@github.com:用户名/gitlearn.git

如果这时没有任何提示说明关联成功了。

3. 推送本地仓库项目到远程仓库
$ git push origin master
Counting objects: 6, done.
Delta compression using up to 4 threads.
Compressing objects: 100% (2/2), done.
Writing objects: 100% (6/6), 412 bytes | 0 bytes/s, done.
Total 6 (delta 0), reused 0 (delta 0)
To git@github.com:liuhaobj/gitlearntest.git
* [new branch]      master -> master

这时已经将本地的项目推送到了远程。刷新github的仓库界面能看到git本地仓库的readme.txt了。

4. 拉取/抓取远程仓库修改

现在我在github添加一个新文件modify.txt,然后执行下面的命令：

$ git fetch origin master
remote: Counting objects: 3, done.
remote: Compressing objects: 100% (2/2), done.
remote: Total 3 (delta 0), reused 0 (delta 0), pack-reused 0
Unpacking objects: 100% (3/3), done.
From github.com:liuhaobj/gitlearntest
* branch            master     -> FETCH_HEAD
306ecb9..4aaf6ee  master     -> origin/master

这时我执行的拉取操作已经成功了，但是当你去gittest文件夹内并未发现modify.txt文件，这就是拉取fetch和抓取pull的区别，fetch并不会执行自动合并远程修改的内容到本地仓库，这时我再执行pull命令

$ git pull origin master
From github.com:liuhaobj/gitlearntest
* branch            master     -> FETCH_HEAD
Updating 306ecb9..4aaf6ee
Fast-forward
modify.txt | 1 +
1 file changed, 1 insertion(+)
create mode 100644 modify.txt

这时再去gittest下就能看到modify.txt文件了.

5. 远程仓库其他操作
  • git remote show [origin],查看远程仓库详细信息
  • git remote rename [原分支名][新分支名] 给分支重命名
  • git remote rm [branch name] 移除分支
  • git remote 查看远程仓库名，[-v]远程仓库Url

分支管理

git的分支管理优点和作用这种理论知识不在此多讲，零基础同学可转向廖雪峰的git教程,直接奔命令行了。

1. 创建新分支Ver01
$ git checkout -b ver01
Switched to a new branch 'ver01'
$ git branch
master
* ver01

可以看到ver01创建成功了，并且当前分支在ver01分支上,-b相当于git branch ver01和git checkout ver01两行命令.这时你可以在ver01分支上工作了。一个功能在ver01上开发完成后可能会合并到master分支上。

2. 分支合并

现在我在ver01分支上对readme.txt文件添加了一句话,提交并切换到master分支上

$ git add readme.txt
$ git commit -m "ver01作了修改"
[ver01 0ffbb8a] ver01作了修改
1 file changed, 2 insertions(+), 1 deletion(-)
$ git checkout master
Switched to branch 'master'

此时去gittest文件夹发现readme.txt是没有修改的,因为我们只对ver01分支的readme文件做了修改。下面将ver01分支合并到mster分支上：

$ git merge ver01
Updating 4aaf6ee..0ffbb8a
Fast-forward
readme.txt | 3 ++-
1 file changed, 2 insertions(+), 1 deletion(-)

这是发现readme.txt文件是修改过的文件，分支合并成功了。

3. 删除分支
$ git branch -d ver01
Deleted branch ver01 (was 0ffbb8a).
$ git branch
* master

这时分支ver01被成功删除了。如果此时ver01分支有新功能开发完成后为合并到master分支此时删除会失败，如果想强制删除可用 git branch -D ver01命令。

4. 分支合并冲突问题

当多人开发时经常会遇到合并冲突问题，比如这样:新建ver02分支，修改readme.txt文件提交，切换到master分支，同样修改readme.txt文件，然后合并ver02分支到master分支

$ git merge ver02
Auto-merging readme.txt
CONFLICT (content): Merge conflict in readme.txt
Automatic merge failed; fix conflicts and then commit the     result.

这时会提示我们合并失败，打开readme.txt文件，如下图


这时需要手动解决冲突,删除readme.txt文件你不想要的内容，然后提交，这时就合并成功了。可以通过git log –graph –pretty=oneline –abbrev-commit命令查看分支合并记录

$ git log –graph –pretty=oneline –abbrev-commit
65182aa 解决冲突
|\
| c83c84f 我在ver02分支上修改了readme.txt
| f611601 我在master分支上修改了readme.txt
|/ 0ffbb8a ver01作了修改
4aaf6ee create modify.txt 306ecb9 def commit
b50ea8f abc commit
###### 5. 存储当前修改git stash
如果此时正在开发一个功能，代码写了一半，领导现在让你用另一个版本打个包或者现在发现了一个bug需要新建个hotfix分支修复bug，但此时你正在开发的功能还没完成，又不想提交到git仓库，这是就可以用到stash命令把当前修改暂时存储起来。

git stash
Saved working directory and index state WIP on master: 65182aa 解决冲突
HEAD is now at 65182aa 解决冲突

$ git status
On branch master
nothing to commit, working directory clean
我先修改了readme.txt文件，然后执行stash，通过status发现当前分支已经没有可提交修改了。这是再新建一个hotfix分支修改bug或者打包等工作。

$ git checkout -b hotfix
Switched to a new branch ‘hotfix’

$ git branch hotfix
master
ver02
这时我在hotfix分支修改readme.txt文件并提交

$ git status
On branch hotfix
Changes not staged for commit:
(use “git add …” to update what will be committed)
(use “git checkout – …” to discard changes in working directory)

modified: readme.txt

no changes added to commit (use “git add” and/or “git commit -a”)

$ git add readme.txt
$ git commit -m “我在hotfix修改了一个bug”
[hotfix 4fb6dfe] 我在hotfix修改了一个bug
1 file changed, 1 insertion(+)
$ git status
On branch hotfix
nothing to commit, working directory clean
至此 hotfix的bug修复完成，然后切换到master分支继续之前的功能开发,并用git stash pop命令恢复之前的修改，并删除存储的内容

$ git checkout master
Switched to branch ‘master’

$ git status
On branch master
nothing to commit, working directory clean

$ git stash list
stash@{0}: WIP on master: 65182aa 解决冲突

$ git stash pop
On branch master
Changes not staged for commit:
(use “git add …” to update what will be committed)
(use “git checkout – …” to discard changes in working directory)

modified: readme.txt

no changes added to commit (use “git add” and/or “git commit -a”)
Dropped refs/stash@{0} (086708f213d325e50451d1676185de6b7b7a42d5)

git stash list查看保存的记录，你可以多次保存，然后恢复指定的保存版本，也可以用git stash apply恢复，但不会删除存储位置的内容,需要再次调用git stash drop进行删除。

##远程分支
######1. 推送本地分支到远程git push origin [branchname]
先创建一个本地分支ver03，修改readme.txt内容提交，然后推送到远程分支，如下：

$ git checkout -b ver03
Switched to a new branch ‘ver03’
$ git add readme.txt
$ git commit -m “ver03 分支”
[ver03 680e5f4] ver03 分支
1 file changed, 1 insertion(+)

$ git push origin ver03
Counting objects: 15, done.
Delta compression using up to 4 threads.
Compressing objects: 100% (14/14), done.
Writing objects: 100% (15/15), 1.50 KiB | 0 bytes/s, done.
Total 15 (delta 2), reused 0 (delta 0)
remote: Resolving deltas: 100% (2/2), done.
To git@github.com:liuhaobj/gitlearntest.git
* [new branch] ver03 -> ver03
此时可以去自己的github仓库看到ver03分支了，如下图：
2. 跟踪远程分支

如果你的git版本小于1.6.2，可以用这个命令git checkout -b [brnach name] [remote name]/[remote branch name],可以在本地生成一个跟踪远程分支的本地分支

如果你的git大于1.6.2版本，可以直接用以下命令

$ git checkout --track origin/ver04
Branch ver04 set up to track remote branch refs/remotes/    origin/ver04.
Switched to a new branch "ver04"

此时在本地会生成一个新的ver04分支，当然你想用不同于远程的分支名，可以用第一种方法生成。

3. 删除远程分支 git push [remote name] –delete [branch name]
$ git push origin --delete ver03
To git@github.com:liuhaobj/gitlearntest.git
- [deleted]         ver03

这是github上的ver03分支就被删除了。

打标签

当某一个版本节点特别重要时，比如需要上线的版本，这时你可以给这个版本打一个标签；git的标签有两种轻量标签（lightweight）与附注标签（annotated）;所谓轻量标签就是你某次提交的一个引用，而附注标签是存储在 Git 数据库中的一个完整对象。 它们是可以被校验的，它可以包含你的电子邮箱，日期等打标签者的信息。

轻量标签
$ git status
On branch ver03
nothing to commit, working directory clean
$ git tag v1.0.0
$ git tag
v1.0.0

如上我给ver03分支打了一个名为v1.0.0的轻量标签，并可以用git tag查看当前标签列表,这时你可以用git show [tag name]查看此标签版本信息

附注标签
$ git checkout hotfix
Switched to branch 'hotfix'
$ git tag -a v1.1.0 -m "this is hotfix version v1.1.0"
$ git tag
v1.0.0
v1.1.0

$ git show v1.1.0
tag v1.1.0
Tagger: xxxxxx <xxxxxxxxx@163.com>
Date:   Thu Aug 3 09:54:04 2017 +0800

this is hotfix version v1.1.0

commit 4fb6dfe3449e8b2175a8f992508c02aa5846d669
Author: liuhao <15811481720@163.com>
Date:   Wed Aug 2 14:48:22 2017 +0800

我在hotfix修改了一个bug

diff --git a/readme.txt b/readme.txt
index 2f71757..1afbee9 100644
--- a/readme.txt
+++ b/readme.txt
@@ -2,3 +2,4 @@ def
  我在ver01分支上作了修改
  我在master分支上修改了

+我在hotfix分支修改了一个bug

可以看到我给hotfix分支打了一个v1.1.0的标签，并给了一个标签信息（-m后面内容），通过git show [tag name]可以查看v1.1.0版本的相关信息

历史版本打标签
$ git log --pretty=oneline

查看历史版本信息

$ git tag -a v1.2 ab23cdd90

给commit id为 ab23cdd90开头的版本打tag

共享标签
$ git push origin v1.0.0

推送v1.0.0版本到远程仓库

$ git push origin --tags

推送本地仓库所有远程仓库没有的标签
